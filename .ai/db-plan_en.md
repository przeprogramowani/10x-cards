## 1. Tables

### Table: users
Managed by Supabase.

### Table: flashcards
Stores flashcards (both manually added and generated by AI).

- id (BIGSERIAL, PRIMARY KEY)
- user_id (UUID, NOT NULL)
  - Foreign key (FK) ‚Üí users.user_id
- question (VARCHAR(200), NOT NULL)
- answer (VARCHAR(500), NOT NULL)
- source (VARCHAR, NOT NULL)
  - Expected values: ‚Äòmanual‚Äô, ‚Äòai-full‚Äô, ‚Äòai-edited‚Äô.
  - CHECK CONSTRAINT may be used
- created_at (TIMESTAMP WITH TIME ZONE, NOT NULL, DEFAULT NOW())
- updated_at (TIMESTAMP WITH TIME ZONE, NOT NULL, DEFAULT NOW())
- generation_id (BIGINT, NULLABLE)
  - Foreign key (FK) ‚Üí generations.id
  - on update cascade, on delete set null

Constraints:
1. CONSTRAINT flashcards_question_length CHECK (char_length(question) <= 200)
2. CONSTRAINT flashcards_answer_length CHECK (char_length(answer) <= 500)
3. CONSTRAINT flashcards_source_enum CHECK (source IN (‚Äòmanual‚Äô, ‚Äòai-full‚Äô, ‚Äòai-edited‚Äô))

### Table: generations
Records details of each successful attempt by the AI to generate flashcards.

- id (BIGSERIAL, PRIMARY KEY)
- user_id (UUID, NOT NULL)
  - Foreign key (FK) ‚Üí users.user_id
- created_at (TIMESTAMP WITH TIME ZONE, NOT NULL, DEFAULT NOW())
- model_type (VARCHAR, NOT NULL)
- number_of_suggestions (INTEGER, NOT NULL)
- source_text_hash (VARCHAR, NOT NULL)
- source_text_length (INTEGER, NOT NULL)
- generation_time (INTEGER, NOT NULL)
  - Time (in milliseconds) needed for generation
- accepted_flashcards_unedited (INTEGER, NULLABLE)
- accepted_flashcards_edited (INTEGER, NULLABLE)

### Table: generation_error_logs
Stores information about errors that occurred when the AI generated the flashcards.

- id (BIGSERIAL, PRIMARY KEY)
- user_id (UUID, NOT NULL)
  - Foreign key (FK) ‚Üí users.user_id
- created_at (TIMESTAMP WITH TIME ZONE, NOT NULL, DEFAULT NOW())
- model_type (VARCHAR, NOT NULL)
- source_text_hash (VARCHAR, NOT NULL)
- source_text_length (INTEGER, NOT NULL)
- generation_time (INTEGER, NOT NULL)
- error_code (VARCHAR, NOT NULL)
- error_message (TEXT, NOT NULL)

## 2. Relationships between tables

- Table `users` ‚Üí Table `flashcards` relationship 1:N.
  - Each user can have multiple flashcards.
- Table `users` ‚Üí Table `generations` relationship 1:N
  - Each user can have multiple records in the generations table.
- Table `users` ‚Üí Table `generation_error_logs` relationship 1:N
  - Each user can have multiple records in the error logs.
- Table `generations` ‚Üí Table `flashcards` relationship 1:N.
  - Each generation record can produce many flashcards.

## 4. PostgreSQL rules (RLS)

As assumed in the notes, with Row-Level Security (RLS) enabled, policies can be set to restrict access to rows based on `user_id`. Example configuration:

``sql
-- Enable RLS
ALTER TABLE flashcards ENABLE ROW LEVEL SECURITY;

-- Policy to allow user to access only their own records
CREATE POLICY select_own_flashcards ON flashcards
  FOR SELECT
  TO public
  USING (auth.uid() = user_id);

-- insert policy: user can insert a record when user_id is the same as current_user_id
CREATE POLICY insert_own_flashcards ON flashcards
  FOR INSERT
  TO public
  WITH CHECK (auth.uid() = user_id);

-- Analogous policies for update and delete
CREATE POLICY update_own_flashcards ON flashcards
  FOR UPDATE
  TO PUBLIC
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY delete_own_flashcards ON flashcards
  FOR DELETE
  TO public
  USING (auth.uid() = user_id);
```

Identical rules should be implemented in the `generations` and `generation_error_logs` tables.

## 5. Additional notes

1. the user identifier (`user_id`) in the `flashcards`, `generations` and `generation_error_logs` tables should be consistent with the value managed by the auth Supabase mechanism.
2. the default timestamp values (`DEFAULT NOW()`) shall ensure that the time of the event is recorded without additional operations from the application.
3. the `source` column in the `flashcards` table makes it possible to distinguish between manually created and AI-sourced (complete or edited) fiches.
4. character limits (200 and 500) in the `flashcards` table have been enforced by CHECK CONSTRAINT - these can also be checked on the application side in forms.
5. care should be taken to regularly clean up the `generation_error_logs` tables if the number of errors is sometimes high (e.g. through archiving or retention mechanisms).
